1.5 Glossary: Terms & How They Relate

Before we dive deeper — here's every term, its definition, and how it connects to the others.

TermWhat it isAnalogy
Episode One complete shopping attempt (navigate → search → evaluate → buy). We run 10 total. Like one game in a 10-game series. Same goal every time, different approach.
Strategy The game plan for one episode. Picked by the RL bandit. Has 4 dimensions (site, query, filter, abandon). Like picking which store to visit + how to search.
RL Bandit The "coach" that picks the strategy each episode. It's actually 4 independent bandits (one per dimension). Like a coach who picks plays. Gets better as it sees which plays score.
Bandit One decision-maker for one dimension (e.g., "which site?"). Contains multiple arms. Like a slot machine with multiple levers. You learn which lever pays best.
Arm One option within a bandit. E.g., "stockx" is an arm of the site bandit. Has a Beta distribution that tracks how good it is. One lever on the slot machine. Pull it, get a reward (or not).
Thompson Sampling The algorithm that picks which arm to pull. Draws a random sample from each arm's distribution, picks the highest. Like asking each option "how good are you?" and believing the most confident answer.
Beta Distribution Each arm tracks successes (alpha) and failures (beta). More data → narrower distribution → more confident. A "confidence curve." Starts flat (no idea), becomes peaked (very sure).
ReAct Loop How Claude shops within one episode: Think → Act → Observe → repeat. Each iteration is one "step." Like thinking out loud while shopping. "Hmm, try this... nope... try that... found it!"
Step One action inside an episode (NAVIGATE, SEARCH, EVALUATE, PURCHASE, etc.). Fewer steps = higher reward. Each click/action you'd take on a website.
Reward The score for one episode (0-99 pts). Based on: did you buy? right product? how many steps? Your grade on the shopping attempt.
Reflexion After each episode, Claude writes a one-sentence lesson in plain English. Stored in memory for future episodes. "Note to self: StockX has cheap Nike shoes."

The Complete Hierarchy (How Everything Nests)

SYSTEM HIERARCHY: Run → Episodes → Strategy → Steps
How these relate (read this carefully)

1 Full Run = 10 episodes
1 Episode = 1 strategy (from bandit) + 1 ReAct loop (Claude shopping) + 1 reward score
1 Strategy = 4 choices (one from each bandit): site + query + filter + abandon
1 Bandit = multiple arms. Thompson Sampling picks one arm per episode.
1 Arm = one option (e.g., "stockx"). Has a Beta(alpha, beta) that gets smarter after each use.
1 ReAct Loop = 4-12 steps where Claude thinks + acts + observes until it buys or gives up
1 Step = one action (NAVIGATE, SEARCH, EVALUATE, PURCHASE, ASK_MEMORY, ABANDON)

This is where Claude actually shops. Each iteration: Think → Act → Observe → repeat until done.

Example: Slow Run (9 steps, 64 pts)

Episode 1 — GOAT (exploring) 9 steps
THOUGHT 1 No memory yet. Let me try GOAT as the bandit suggests.
ACTION 1 NAVIGATE [https://www.goat.com]
OBSERVE 1 Page loaded. Ready to search.
ACTION 2 SEARCH [Nike Adidas sneakers]
OBSERVE 2 5 products found. ALL over $120 budget!
THOUGHT 3 Everything on GOAT is over budget. I need to abandon and try another site.
ACTION 3 ABANDON
ACTION 4 NAVIGATE [https://stockx.com]
ACTION 5-6 SEARCH → EVALUATE
OBSERVE Found $105 Nike, all criteria match.
ACTION 7-9 PURCHASE → Done (extra steps from ASK_MEMORY + evaluation)
PENALTY 9 steps × -4 = -36 pts (vs only -16 for 4 steps)
Why does the curve go up?

Episode 1: Agent doesn't know which site is best → tries GOAT (expensive) → abandons → finds product on StockX → 9 steps, 64 pts.
Episode 10: Agent already knows StockX works → goes directly there → buys immediately → 4 steps, 99 pts.
The step penalty (-4/step) punishes inefficiency. The efficiency bonus (+15 for ≤4 steps) rewards directness. Together they create the upward curve.

4. Thompson Sampling (The RL Engine)

How the bandit decides which strategy to try each episode.

THOMPSON SAMPLING IN 4 STEPS

Why Thompson Sampling is perfect here

  • Early episodes: All arms have wide distributions (Beta(1,1) is uniform). Random samples are all over the place → agent explores different sites.
  • Later episodes: Winning arms have peaked distributions (Beta(5,2) peaks around 0.7). They consistently sample high → agent exploits what works.
  • No tuning: Unlike epsilon-greedy (need to set epsilon), Thompson Sampling figures out explore/exploit automatically.
  • Works with 10 samples: Deep RL (PPO, DQN) needs thousands of episodes. Thompson Sampling converges in 3-5.

The design insight

7. The Hallucination Fix (Critical Bug We Solved)

Without this fix, 6/10 episodes produced 0 reward because Claude "shopped" in its imagination.

Before Fix (Broken)

Thought: I'll search for shoes Action: SEARCH nike shoes Observation: Found 3 products: ← HALLUCINATED [0] Nike Pegasus | $99 ← NOT REAL [1] Nike Vomero | $149 ← MADE UP Thought: Let me buy #0 Action: PURCHASE 0 Observation: Purchase confirmed! ← FAKE Done: Bought Nike Pegasus for $99 ← NEVER HAPPENED

Problem: Claude generates the entire Thought/Action/Observation sequence in one shot. The real environment is never called. The agent "shops" in its imagination.

After Fix (Working)

Thought: I'll search for shoes Action: SEARCH nike shoes [STOP] ← Claude is forced to stop here # System calls real environment # Jina.ai fetches stockx.com # Returns REAL products with REAL prices Observation: Found 5 products: ← REAL DATA [0] Nike Griffey Max 1 | $105 ← FROM STOCKX [1] Nike AF1 Low | $75 ← ACTUAL PRICE

Fix: stop_sequences=["Observation:"] in the API call. Claude literally cannot generate "Observation:" — the API stops it, returns control to us, we call the real environment, then feed the real observation back.

The one line that fixed everything
response = client.messages.create( model="claude-sonnet-4-6", max_tokens=600, stop_sequences=["Observation:", "observation:"], # THIS LINE system=build_system_prompt(context_graph, history_summary), messages=messages, )

8. Agent-to-Agent Communication

The Shopper Agent can query the Memory Agent mid-run. This is a real inter-agent protocol.

ASK_MEMORY PROTOCOL (mid-episode communication)
Why this matters for "Internet of Agents"

This isn't just sequential handoff (Agent A finishes → Agent B starts). The Shopper Agent chooses to query Memory during its reasoning loop when it needs information. It's a synchronous request/response protocol between two running agents. This is the "agent-to-agent" theme the hackathon is about.

9. Safety: Auth + Payment Chain

How we ensure the agent can't overspend or buy from unauthorized sites.

PAYMENT AUTHORIZATION FLOW
The safety invariant

The credential is issued ONCE at episode start. It cannot be modified mid-run. Even if Claude is "jailbroken" and tries to spend $500 on a shady site, the Payment Agent's verification will reject it. No agent can escalate its own permissions.

10. Design Decisions (For Judge Q&A)

Why we chose X over Y. Good ammunition for defending the project.

DecisionAlternativeWhy We Chose This
Thompson Sampling Epsilon-greedy, UCB, PPO/DQN No hyperparams. Natural explore/exploit. Converges in 3-5 runs (deep RL needs thousands).
4 decomposed bandits Single bandit (216 arms) 16 total arms vs 216. Each gets 2-3 updates in 10 runs. Mathematically sound.
Jina.ai reader Playwright, Selenium Free, no browser needed. 10 episodes in 2 min vs 20+ with real browsers.
ReAct loop Structured tool-use Judges can read Claude's reasoning. More impressive. Shows genuine thought.
stop_sequences Post-hoc validation Prevents hallucination at source. API-level guarantee, not hope.
Auth before shopping Check only at payment Defense in depth. Credential is immutable — can't be jailbroken mid-run.
Reflexion lessons Pure numeric signal Interpretable. "Avoid GOAT for budget shoppers" is readable. Also feeds into future system prompts.
Step penalty -4/step Flat reward Creates visible learning curve. Without it, all successful runs score the same.
Live + simulated hybrid All one or the other 4 live sites prove it's real. 2 simulated show graceful failure handling.
